Using the API

Carbon is API-first — create a key and make your first call.

Everything you can do in the Carbon web app — create a quote, release a job, raise a purchase order, read a part's method — you can also do over the API (Application Programming Interface): a way for your own code or another program to talk to Carbon directly. Carbon is API-first, which means the app you click through and the API speak to the same backend. There's no second-class "integration layer" bolted on later; the buttons in the UI call the exact same endpoints you're about to call yourself.

This page gets you from zero to a working call: you'll create a key, pick an official client library, and run a few lines of code that read real data back out of Carbon. It's a friendly on-ramp — the full endpoint-by-endpoint reference lives in the separate Developer Docs, which we'll link as we go.

Where this fits. This is the first stop in the Developing track. Once you can make a call, the next pages build on it: AI & the MCP server lets AI assistants drive Carbon for you, Integrating with other systems connects Carbon to your other tools, and Migrating data uses the same API to bulk-load your existing records. If you've never touched an API before, that's fine — every step below is spelled out.

What "API-first" means for you

A quick mental model before you write any code. When you confirm a sales order in the web app, your browser sends a request to a Carbon endpoint (a specific URL that performs one action), and Carbon sends back a response. The API just lets your program send that same kind of request. So anything you've read about in this manual — items, quotes, sales orders, jobs, purchase orders, receipts, shipments, invoices, quality records — is reachable as data you can read and (with the right key) write.

If it's in the app, it's in the API. There is no feature that's "UI only." The product team builds the UI on top of the public API, so the moment a capability ships in Carbon it's callable from your code too. That's the single most useful thing to know on this page.

Notice that "Your code" and "Carbon web app" point at the same backend. That's the whole idea: the API isn't a copy of your data, it is your data.

Create an API key

Before any program can talk to Carbon, it needs to prove who it is. The simplest way is an API key — a long secret string that stands in for your login. You create it once, store it safely, and send it with every request.

Open API Keys in Settings

In the Carbon web app, go to Settings → API Keys. This page lists every key that exists for your company, who created it, and when it was last used. New companies start with none.

Create a new key

Click New API Key, give it a clear name that says what it's for (for example, Shipping integration or Nightly export), and create it. Naming keys by purpose makes it painless to revoke the right one later without breaking everything else.

A key inherits the permissions of the user who created it. If your login can't approve purchase orders in the app, a key you create can't either. That's a feature — it means a key is never more powerful than the person behind it. See Users & permissions for how those roles work.

Copy the secret immediately

Carbon shows you the full key value exactly once, right after you create it. Copy it now and store it somewhere safe — a secrets manager, your CI provider's encrypted variables, or a password manager. If you lose it, you can't view it again; you simply delete the key and make a new one.

Treat a key like a password. Anyone holding it can act as you in Carbon. Never paste a key into client-side code, a public repository, a screenshot, or a support ticket. Keep it in an environment variable (the examples below read CARBON_API_KEY), and revoke any key the moment you suspect it leaked — deleting it on the API Keys page kills it instantly.

API key vs. OAuth — which auth do you need?

"Auth" (authentication) is just how Carbon knows who is calling. Carbon supports two styles, and the right one depends on whose data you're acting on.

An API key authenticates you (or a single service account). You create it once, store the secret, and send it on every request. This is the right choice for the work most readers of this manual are doing:

  • A script that exports your own orders every night.
  • A backend integration that pushes shipments into Carbon.
  • One-off data migration or reporting jobs.

Send it as a request header. Carbon also reads your company automatically from the key, so you don't pass a company ID by hand:

OAuth is for apps that act on behalf of other Carbon users — for example, a third-party product you're building that lets its users connect their Carbon account. Instead of holding a static secret, your app sends the user through a consent screen and receives a short-lived access token scoped to that user's permissions.

If you're automating your own company, you almost certainly want an API key and can skip OAuth for now. If you're building a public integration that other companies install, the OAuth flow (client registration, scopes, token refresh) is documented in full in the Developer Docs.

Rule of thumb: automating your Carbon data → API key. Building an app other Carbon customers log into → OAuth. Either way, the request examples below are identical once you have a token; only how you obtain it differs.

Use an official client library

You can call Carbon with raw HTTP (any fetch or curl will do), but the team maintains official client libraries that wrap the endpoints in typed, autocompleting functions and handle auth headers, pagination, and errors for you. Reach for these first.

TypeScript / JavaScriptnpm i @carbon/apiWeb apps, Node services, edge functions
Pythonpip install carbon-apiData work, scripts, notebooks, ETL
.NETdotnet add package Carbon.ApiC# services and desktop tools

The exact package names and the always-current install commands are in the Clients section of the Developer Docs — check there if an npm/pip name differs from what you see above, since published names can change between versions.

Make your first call

Time to read real data. The goal of a first call is simply to prove that your key works and you can get a response — so we'll read, not write. Read before you write. Carbon's read endpoints can't change anything, so they're the safe place to learn the data shapes. Once you're comfortable, the same client exposes create/update methods (for example, creating a quote line or releasing a job) — guarded by your key's permissions. When you're ready to write, do it against a test or demo company first, never your live data.
