AI & the MCP server

Let AI assistants work in Carbon through the Model Context Protocol.

Carbon ships with a built-in MCP server — a standard way to let an AI assistant read and act inside your Carbon company using plain English. Instead of clicking through screens, you can ask Claude, ChatGPT, or your code editor "what's the on-hand quantity of A36 steel?" or "release every sales order stuck in Needs Approval for three days" — and the assistant does it by calling the same Carbon API the app itself uses. This page explains what MCP is, how Carbon's server is organized, how to keep it safe, and how to connect an assistant.

Where this fits. This is the second of the three Developing pages. The API is the raw machinery — every action in Carbon as a callable endpoint. MCP wraps that same machinery so an AI assistant can drive it for you. If you'd rather wire Carbon to another system than talk to it, jump to Integrating with other systems.

What is MCP?

MCP stands for the Model Context Protocol — an open standard that lets AI assistants connect to outside tools and data in a consistent way. Think of it like a USB port for AI: any assistant that "speaks MCP" can plug into any system that exposes an MCP server, with no custom glue code in between.

A Carbon MCP server is just a program that lists the things an assistant is allowed to do (these are called tools) and runs them on request. When you ask a connected assistant a question, it picks the right Carbon tools, fills in the details, and reports back — reading records, drafting documents, or making changes, depending on what you've allowed.

Carbon's server exposes roughly 1,188 tools spanning 15 modules — Sales, Purchasing, Items, Production, Inventory, Quality, Accounting, Resources, and more. In practice that means nearly anything you can do by clicking in Carbon, an assistant can do through MCP.

Carbon hosts this for you at app.carbon.ms/mcp. The server you connect to is the same one Carbon's own AI features use — there's nothing to install or run on your own machine.

The three meta-tools

A thousand-plus tools is far too many to hand an assistant all at once — it would overwhelm the model's context before it answered a single question. So Carbon's server hides the full catalog behind just three "meta-tools" that the assistant uses to discover and run everything else on demand.

Meta-toolWhat it doesExample
search_toolsFinds the right tools for a task by keyword, module, or safety class.search_tools({ query: "customer" })
describe_toolReturns the exact inputs and outputs (the schema) for one tool.describe_tool({ name: "sales_getCustomers" })
call_toolRuns a tool with its parameters and returns the result.call_tool({ name: "sales_getCustomers", arguments: { args: { limit: 10 } } })

The flow is always the same: search, then describe (optional), then call. Suppose you ask "what's the on-hand quantity of A36 steel?" Behind the scenes the assistant:

Searches for the right tool. It calls search_tools({ query: "inventory quantity" }) and gets back candidate tool names like inventory_getItemQuantities.

Checks the inputs. If it's unsure what parameters the tool needs, it calls describe_tool({ name: "inventory_getItemQuantities" }) to read the schema — which fields are required, what shape the response takes.

Calls the tool. It runs call_tool with the item ID for A36 Steel Plate (A36-STEEL-PLT-1".96x240) and reads the result back to you in a sentence: "There's none on hand at Headquarters right now — every pound is committed to open jobs."

You never see these three steps — you just see the answer. But knowing the pattern helps you understand what your assistant is doing and why an answer occasionally takes a couple of round-trips.

Every result comes back in the same envelope: { data, error?, count? }. The assistant reads data for the answer, checks error if something went wrong, and uses count to know how many records matched. IDs like companyId are filled in automatically from your credentials, so you never have to specify which company you're working in.

Tool safety classes

Not every tool is equal. Reading the on-hand quantity of steel is harmless; deleting a sales order is not. So Carbon tags every tool with one of three safety classifications, and you can filter by them — for example, search_tools({ classification: "READ" }) to surface only the look-but-don't-touch tools.

ClassWhat it can doExample toolsRisk
READLooks up and returns data. Changes nothing.sales_getCustomers, inventory_getItemQuantitiesNone
WRITECreates or updates records — new quotes, edited orders, released jobs.sales_createQuote, production_updateJobReversible
DESTRUCTIVEDeletes records or makes changes that are hard to undo.sales_deleteSalesOrderPermanent

Treat WRITE and DESTRUCTIVE tools like a teammate with edit access. An assistant calling them is making real changes to your live company data — the same as if a colleague clicked the buttons. When you're exploring or learning, keep your assistant to READ tools only. Turn on WRITE access deliberately, for a specific task, with a human reviewing the result.

What you can actually ask

The point of MCP isn't to memorize tool names — it's to describe what you want in plain language and let the assistant find the tools. Here are realistic asks against Acme's data, and roughly what happens under the hood.

"What's the on-hand quantity of A36 steel?"

A pure READ request. The assistant searches for an inventory tool, calls it with the A36 plate's item ID, and answers in a sentence. Nothing changes. Other safe reads:

  • "Show me every job for Radiant Aerospace that isn't finished yet."
  • "What did we invoice Radiant on AR000025, and is it paid?"
  • "List open purchase orders to Ryerson."

"Which sales orders have been stuck in Needs Approval for three or more days?"

The assistant reads the sales-order list (a READ call), filters by status and date, and hands you the offenders. This is the sweet spot for MCP — turning a question that would take several filtered screens into one sentence. Similar triage asks:

  • "Any purchase orders sitting in Draft older than a week?" (PO000056 to Ryerson would surface.)
  • "List jobs past their due date that aren't complete."
  • "Which invoices are overdue and unpaid?"

"Release every sales order stuck in Needs Approval for three or more days."

Now the assistant first reads to find the stuck orders, then issues a WRITE call per order to release each one. Because these tools change live data, you'll want to have approved WRITE access and to review the list before it acts. Other action asks:

  • "Create a quote for 500 Counterweight Hooks for Radiant Aerospace at $73.14 each."
  • "Bump the priority on job J000099 to the top of the queue."

Good prompts are specific. "Release the stuck orders" is fine; "Release sales orders in Needs Approval with an order date 3+ days ago, and show me the list first" gives the assistant a clean target and you a checkpoint before anything changes.

How the assistant reaches Carbon

When you connect an assistant, two things happen on every request: the assistant authenticates as a real Carbon user, and Carbon enforces that user's permissions on every tool call.

Loading diagram...
  • Authentication uses either an OAuth sign-in (you log in to Carbon once, the assistant gets a scoped token) or an API key you generate in Carbon's settings — the same key you'd use for the API.
  • Permissions are inherited from whoever the credentials belong to. If a user can't delete sales orders in the app, the assistant can't either — even if it tries the DESTRUCTIVE tool. MCP is a new doorway into Carbon, not a way around its permission model.

An API key is a password. Anyone holding it can act as that Carbon user through MCP. Store keys in your assistant's secret settings (never paste them into a chat), give each integration its own key, and revoke a key the moment it's no longer needed.

Connect an assistant

The exact menu differs by assistant, but the shape is identical everywhere: point the tool at Carbon's MCP URL and give it credentials. Here's the flow for a desktop AI client.

Generate credentials in Carbon. Sign in to Carbon, open Settings → API keys, and create a key (or set up OAuth if your assistant supports it). Copy the key somewhere safe — you won't be able to see it again.

Add Carbon as an MCP server. In your assistant's settings, find the connectors or MCP section and add a new server pointing at app.carbon.ms/mcp. Paste your API key or complete the OAuth sign-in when prompted.

Confirm the connection. The assistant should now list Carbon among its available tools. You'll typically see the three meta-tools — search_tools, describe_tool, call_tool — which is exactly what you want; the full catalog stays hidden behind them.

Start with a READ question. Ask something safe first, like "What's the on-hand quantity of A36 steel?" or "List Radiant Aerospace's open jobs." If you get a sensible answer, you're connected. Only then move on to WRITE-level asks, deliberately.

The call_tool meta-tool inherits the safety class of whatever tool it runs. Calling inventory_getItemQuantities through it is a READ; calling sales_deleteSalesOrder through it is DESTRUCTIVE. The wrapper doesn't soften the underlying risk — so the READ-first habit matters even though call_tool itself looks generic.
